## Epic

**Epic Name**: platform — multi-tenancy

**Description**:

soorma-core currently operates on a single-tier tenancy model where a UUID `tenant_id` identifies every tenant uniformly across all services. This creates a fundamental architectural limitation: it conflates the *platform tenants* (engineering teams building agentic systems on soorma-core) with the *service tenants* (the end-customers of those agentic systems). This initiative replaces the existing model with a **two-tier tenancy design** that clearly separates platform-level identity (authenticated via HTTP headers from soorma-core's auth channel) from service-level identity (trusted strings supplied by the authenticated platform tenant's own systems).

The implementation spans the full soorma-core stack: shared libraries (`soorma-common`, `soorma-service-common`), all backend services (registry, memory, tracker, event-service), and the Python SDK. A key deliverable is the activation of PostgreSQL Row-Level Security in the memory service — previously configured in the schema but never enforced at runtime — via a shared `TenancyMiddleware` that populates session variables used by RLS policies on every database request.

**Acceptance Criteria**:
- All tenant and user identity columns across registry, memory, and tracker services store opaque strings (max 64 chars) rather than UUID types, with Alembic migrations that require no data migration for the pre-production system
- A new shared `soorma-service-common` library provides `TenancyMiddleware`, RLS-activating `get_tenanted_db`, and a `TenantContext` convenience bundle consumed consistently by all services
- Platform tenant identity is always extracted from the authenticated HTTP channel (`X-Tenant-ID` header) by middleware — never from user-controlled event payloads; the Event Service is the exclusive authority for stamping `platform_tenant_id` on published events
- SDK clients and PlatformContext wrappers are updated to the two-tier parameter model (`platform_tenant_id` at init time, `service_tenant_id`/`service_user_id` per-call) with no exposure of platform tenant identity to agent handler code
- GDPR data deletion interfaces are implemented in memory and tracker services covering platform-tenant, service-tenant, and service-user scopes

### Story: soorma-service-common: Implement shared TenancyMiddleware and RLS activation library

**Description**:
Creates the entirely new `libs/soorma-service-common` library that all backend services will adopt as their shared tenancy infrastructure. The library provides `TenancyMiddleware` for consistent header extraction into `request.state`, the `get_tenanted_db` dependency that activates PostgreSQL RLS session variables (`set_config`) within each database transaction, and a `TenantContext` convenience bundle that combines all three identity dimensions into a single FastAPI `Depends()` call. It also defines the `PlatformTenantDataDeletion` abstract base class used by memory and tracker for GDPR compliance.

**Acceptance Criteria**:
- `TenancyMiddleware` extracts `X-Tenant-ID`, `X-Service-Tenant-ID`, and `X-User-ID` headers into `request.state`; missing `X-Tenant-ID` defaults to `DEFAULT_PLATFORM_TENANT_ID` from `soorma_common`
- `get_tenanted_db` calls `set_config` for all three session variables (`app.platform_tenant_id`, `app.service_tenant_id`, `app.service_user_id`) within the same database transaction before yielding the session, ensuring RLS policies enforce correctly
- `TenantContext` dataclass and `get_tenant_context` FastAPI dependency provide all three identity dims plus a tenanted DB session in one `Depends()` call
- `PlatformTenantDataDeletion` abstract base class defines `delete_by_platform_tenant`, `delete_by_service_tenant`, and `delete_by_service_user` abstract methods

**Linked Epic**: Epic: platform — multi-tenancy

**Dependencies**: soorma-common: Add platform tenancy constant and EventEnvelope field

### Story: memory: Restructure schema to two-tier tenancy and rebuild RLS policies

**Description**:
The Memory Service receives the most extensive changes in this initiative — a breaking schema migration that drops the `tenants` and `users` FK reference tables, adds `platform_tenant_id VARCHAR(64)`, and renames the existing UUID FK columns to `service_tenant_id` and `service_user_id` across all 8 memory tables. Critically, the existing PostgreSQL RLS policies — previously configured but never enforced at runtime — are dropped and rebuilt from scratch using string comparison against the three session variables activated by `get_tenanted_db`. The service also implements the `MemoryDataDeletion` GDPR interface.

**Sub-tasks**:
- Alembic breaking migration: drop `tenants`/`users` tables, add `platform_tenant_id`, rename columns on all 8 tables, drop old RLS policies, apply new string-based RLS policies
- ORM model updates: replace UUID FK columns with `String(64)` plain columns; update service layer and API DTOs
- Replace local `TenancyMiddleware` with shared one; switch all `Depends(get_db)` to `Depends(get_tenanted_db)` via `TenantContext`
- Implement `MemoryDataDeletion` covering all 6 core memory tables

**Acceptance Criteria**:
- Alembic migration runs cleanly and all 8 tables (`semantic_memory`, `episodic_memory`, `procedural_memory`, `working_memory`, `task_context`, `plan_context`, `sessions`, `plans`) have the three-column identity structure
- RLS policies enforce correctly: a query using the wrong `platform_tenant_id` returns zero rows from tenant-scoped tables
- `MemoryDataDeletion.delete_by_platform_tenant`, `delete_by_service_tenant`, and `delete_by_service_user` delete the correct rows in a single transaction
- All existing Memory Service tests are updated and pass with the new field names and types

**Linked Epic**: Epic: platform — multi-tenancy

**Dependencies**: soorma-common: Add platform tenancy constant and EventEnvelope field, soorma-service-common: Implement shared TenancyMiddleware and RLS activation library

### Story: event-service: Inject platform_tenant_id as Event Service trust boundary

**Description**:
Makes the Event Service the sole authority for stamping `platform_tenant_id` on events before they enter the NATS bus. The change registers `TenancyMiddleware` from `soorma-service-common`, updates the `publish_event` route to accept an HTTP `Request` parameter, and injects (overwrites) `event.platform_tenant_id` from authenticated `request.state.platform_tenant_id` before publishing. This ensures that downstream consumers (Tracker, future services) can unconditionally trust `event.platform_tenant_id` as coming from the authenticated channel.

**Acceptance Criteria**:
- `TenancyMiddleware` is registered in the Event Service `main.py`
- `publish_event` route reads `platform_tenant_id` from `request.state` (set by middleware) and overwrites any SDK-supplied value on the outbound `EventEnvelope` before publishing to NATS
- Any `platform_tenant_id` value sent by the SDK in the publish payload is discarded
- Event Service tests verify the injection and overwrite behavior

**Linked Epic**: Epic: platform — multi-tenancy

**Dependencies**: soorma-common: Add platform tenancy constant and EventEnvelope field, soorma-service-common: Implement shared TenancyMiddleware and RLS activation library

### Story: sdk-python: Update SDK clients and PlatformContext to two-tier tenant parameters

**Description**:
Updates `MemoryServiceClient`, `TrackerServiceClient`, and PlatformContext wrappers to the two-tier tenant model. `platform_tenant_id` is set once at client initialization (from the `DEFAULT_PLATFORM_TENANT_ID` constant or `SOORMA_PLATFORM_TENANT_ID` env var) and sent as `X-Tenant-ID` on every request; per-call parameters are renamed to `service_tenant_id` / `service_user_id`. PlatformContext wrappers never expose `platform_tenant_id` to agent handler code — they extract `service_tenant_id` and `service_user_id` from the event envelope. As a final deliverable, `docs/ARCHITECTURE_PATTERNS.md` Section 1 is updated to document the two-tier tenancy model accurately.

**Acceptance Criteria**:
- `MemoryServiceClient` and `TrackerServiceClient` accept `platform_tenant_id` at `__init__` time and include it as `X-Tenant-ID` on every HTTP request; per-call methods use renamed `service_tenant_id` / `service_user_id` parameters sent as `X-Service-Tenant-ID` / `X-User-ID` headers
- PlatformContext wrappers (`context.memory`, `context.bus`) use the renamed per-call parameters; agent handler code never sees or passes `platform_tenant_id`
- All SDK tests updated to the new two-parameter model and pass
- `docs/ARCHITECTURE_PATTERNS.md` Section 1 correctly documents `X-Tenant-ID` as platform tenant, `X-Service-Tenant-ID` as service tenant, `X-User-ID` as service user, and the `platform_tenant_id` injection behavior of the Event Service

**Linked Epic**: Epic: platform — multi-tenancy
